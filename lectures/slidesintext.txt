% Rust - The Really Really Bad
  ** Copy is implicit — the compiler inserts it silently on assignment or pass-by-value;
       no method call.                                                                           ** Clone is explicit — you must call .clone(); it is a bitwise memory copy by default,
    but can be overridden.
  ** Copy requires the type to be entirely stack-resident
      (no heap allocation, no Box, no Vec, no String).
  ** Clone has no such restriction — it can deep-copy heap data, open files,
     allocate new memory.
  ** Copy requires Clone — any type that implements Copy must also implement Clone,
      but not vice versa.
  ** So you have the optimized bit copy messing with the general copy you want.

% Verus Logic - EPR
  ** Specifications are written in a pure mathematical sublanguage —
     spec functions, forall/exists, arithmetic, sets, sequences.
  ** Z3 handles linear arithmetic, arrays, and quantified formulas,
     but quantifier instantiation requires explicit trigger annotations.
  ** Decidability is not guaranteed — hard proofs need hints:
     intermediate assertions, lemma calls, fuel increases.
  ** The proof obligation is: for every execution path,
     the postcondition follows from the precondition and invariants.
  ** https://en.wikipedia.org/wiki/Bernays%E2%80%93Sch%C3%B6nfinkel_class -
    can be translated into EPR, essentially predicate logic.

% Specs, Requires, Ensures — Insertion Sort
  ** spec fn is_sorted defines what sorted means mathematically:
     forall i < j: v[i] <= v[j]
  ** requires states what the caller must prove before calling:
     insertion_sort has no requires — it works on any slice.
  ** ensures states what the function guarantees on return:
       sorted.len() == old(a).len()
       sorted@.to_multiset() == old(a)@.to_multiset()
       is_sorted(sorted)
  ** Loop invariants extend ensures into loop bodies —
     the verifier checks them on entry, each iteration, and exit:
       a@.to_multiset() == old(a)@.to_multiset()
       forall k < l < up: a[k] <= a[l]
  ** old(a) names the value of a on function entry,
     allowing postconditions to reference what changed.

% Proof Vocabulary: assume, admit, proof, assert
  ** admit() tells the verifier to accept the spec/proof.
  ** assume(P) tells the verifier to accept proposition P at one point
     without proof — a targeted hole, tracked as a proof obligation.
  ** proof { ... } is a ghost block: code that runs only in the verifier,
     not at runtime — used for lemma calls, reveals, intermediate steps.
  ** assert(P) demands the verifier prove P at that point in execution;
     fails if Z3 cannot discharge it from current context.
  ** assert(P) by { ... } provides an inline proof of P —
     the block contains the reasoning steps needed for Z3 to succeed.

% Views and the Libraries
  ** A View maps an executable Rust type to a mathematical ghost type:
     Vec<T> views as Seq<T>,  HashSet<K> views as Set<K::V>.
     Specs are written over the view; exec code manipulates the real type.
  ** vstd is Verus's standard library — specs for Vec, Seq, Set,
     Map, Multiset, arithmetic, and common lemmas.
  ** vstdplus is APAS-VERUS's extension library — specs for
     TotalOrder, finite sets, feq (functional equality),
     Arc/RwLock bridges, and thread utilities.
  ** The View abstraction separates the proof from the representation:
     change Vec to BTreeMap and only the view mapping changes,
     not the specs or proofs above it.
  ** Ghost types (Seq, Set, Map) live only in the verifier —
     they have no runtime cost and no runtime representation.

% Views and the Libraries
  ** View maps an executable type to a mathematical ghost type —
     Vec<T> views as Seq<T>, HashSet<K> views as Set<K::V>,
     HashMap<K,V> views as Map<K::V, V::V>. Specs are written over
     the view; the representation can change without touching proofs.
  ** Seq<T> — mathematical sequence: indexed, ordered, finite.
     Supports indexing, slicing, concatenation, multiset conversion.
  ** Set<T> and Map<K,V> — mathematical set and partial function:
     no duplicates, no ordering, finite or infinite, fully extensional.
     Multiset<T> adds multiplicity for counting arguments.
  ** int and nat — unbounded mathematical integers, no overflow.
     Exec code uses u64/usize; specs lift to int/nat for arithmetic proofs.
  ** vstd is Verus's standard library — lemmas and axioms for
     Seq, Set, Map, Multiset, arithmetic, and relations, all broadcast
     so proofs pick them up automatically via trigger groups.
     vstdplus is APAS-VERUS's extension — TotalOrder, feq, Arc/RwLock
     bridges, and finite set utilities not in vstd.

% Wrapping Rust — Giving Specs to Unverified Code
  ** Rust's standard library is unverified — HashMap, HashSet,
     threads, and I/O have no Verus specs out of the box.
  ** external_type_specification adds a spec to an existing Rust type
     without wrapping it — used for types you cannot change.
  ** The wrapper pattern: define a new struct holding the unverified type,
     implement View to give it a mathematical model,
     then implement each method with external_body and a tight ensures.
  ** Example: HashSetWithViewPlus wraps std::collections::HashSet,
     gives it View = Set<Key::V>, and wraps insert/contains/remove
     with external_body so all proof work happens over the Set view.
  ** The goal: push external_body to the boundary,
     keep all algorithmic proof inside verus!.

% Wrapping Rust — Code Examples
  ** Step 1: define a wrapper struct and give it a View
  ``` pub struct HashSetWithViewPlus<Key: View + Eq + Hash> { pub inner: HashSet<Key>,}

  impl<Key: View + Eq + Hash> View for HashSetWithViewPlus<Key> {
      type V = Set<<Key as View>::V>;
      open spec fn view(&self) -> Self::V { self.inner@.map(|k: Key| k@)}
  }
  ```

% Wrapping Rust — Code Examples
  ** Step 2: wrap each method with requires/ensures and prove.
```
 pub fn insert(&mut self, k: Key) -> (inserted: bool)                                            requires obeys_key_model::<Key>(),
  ensures
   self@ == old(self)@.insert(k@),
   inserted == !old(self)@.contains(k@),
  { self.inner.insert(k) }

  pub fn contains(&self, k: &Key) -> (contains: bool)
      requires obeys_key_model::<Key>(),
      ensures  contains == self@.contains(k@),
  { self.inner.contains(k) }
```
  ** All proof work happens over SetKey::V — the mathematical view —
     not over the unverified HashSet internals.

 % Tokenized State Machines — Hance, CMU 2024
  ** Problem: Rust's ownership types handle sequential aliasing well
     but cannot express distributed protocol state across threads —
     who holds the right to increment, which phase a lock is in,
     whether a producer has deposited a value.
  ** A Tokenized State Machine defines protocol state as fields
     with sharding strategies (variable, map, count, storage_option…).
     Transitions and an inductive invariant are proved once, globally.
     Verus auto-generates ghost token types and exchange functions
     so client code manipulates local tokens, not global state.
  ** RwLock in vstd is implemented via a tokenized state machine —
     the lock's internal protocol (unlocked / read-locked / write-locked,
     reader count) is the TSM; RwLockPredicate is the invariant the user
     supplies; the read/write guard tokens are the generated token types.
  ** Key publications:
     Hance et al. "Sharding the State Machine" — OSDI 2023 (primary TSM paper)
     Hance, Howell, Padon, Parno. "Leaf" — OOPSLA 2023 (storage protocols)
     Hance. PhD Thesis, CMU-CS-CS-24-146, 2024 (full formal treatment)

% APAS-VERUS: Clone and Generics in Verus
  ** Rust Clone vs Copy (recap)
     *** Copy: implicit bitwise copy, compiler-inserted, no user code
     *** Clone: explicit .clone() call, can allocate, user-defined
  ** Verus recognizes Copy as a first-class trait (since 2026-03)
     *** Copy bounds propagate through generics correctly
     *** Fn/FnMut/FnOnce also first-class since the same release
  ** Verus does NOT recognize Clone on closures or named fn items
     *** Cannot write: requires F: Clone  (Verus ignores Clone bound)
     *** Cannot Clone a closure variable inside a proof
     *** Workaround: vstdplus helpers clone_fn / clone_fn2 / clone_pred
  ** Generic spec functions require explicit type bounds
     *** Bounds must appear in requires, not just in the Rust signature
     *** Well-formedness predicates must be stated per type parameter
  ** Clone in exec code: full bodies verified
     *** APAS-VERUS pattern: Clone::clone body uses assume(r == old(self)@)
     *** The assume bridges the ghost/exec gap at the one correct site
     *** Callers obtain the equality through clone()'s ensures clause,
         not by assuming it themselves
  ** Generics + RwLock: type bounds must include StTInMtT, 'static
     *** Ghost fields in generic structs block Send/Sync auto-derive
     *** Fixed in Verus #2287 (April 2026): Ghost<A>: Send + Sync

% Veracity: veracity-minimize-proofs: AIs Write Redundant Proofs
  ** AI proof agents produce correct but bloated proofs —
     redundant asserts, unnecessary proof blocks, duplicate
     intermediate steps that Z3 already knew. They verify,
     but they waste solver budget on every subsequent run.
  ** veracity-minimize-proofs tests each assert and proof block
     individually: remove it, re-verify, keep it out if still clean.
     Fully automated. AST-aware — no string hacking.
  ** Result across APAS-VERUS: 22 asserts and 33 proof blocks removed
     in 105 minutes of wall time. 55 redundant proof statements
     eliminated, ~2 minutes of minimizer time per removal.
  ** It only takes out unnecessary asserts/proofs that decrease
     run time and memory.
  ** Each removal compounds: one assert in Chap43 OrderedTableMtEph
     saved 43–104 s of Z3 CPU and up to 89 MB of Z3 RSS per verify run.
     Eight removals in that one file: Z3 RSS −57%.
  ** ROI: 105 minutes of minimizer run time buys permanent savings
     on every future verification pass.

  *** Multiple windows - you can see your scripts run.

   *** It has decided that a single window is all you need.
   *** You can pop open and close some detail.
   *** You can't see thinking.
   *** It is somewhat better inside emacs.
   *** It clears your text in a variety of ways.

% The AI Paired Programming Interfaces
 ** What you want is pretty obvious with experience.
   *** This is text. You want simple fast search and processing
     of your text interaction with the agents.
   *** You want one window with your core interaction. The agent
    tells you what it is doing in it.
   *** You want one to watch your proofs, compile, tests and scripts.
   *** You want one window to have the LLM show you it's thinking.
   *** I learned a ton from watching agents think.
   *** I suspect they are hiding most of their thinking now to
    prevent them from being used to train new AIs.
